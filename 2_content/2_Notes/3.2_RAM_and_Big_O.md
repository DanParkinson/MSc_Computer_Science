## The RAM Model

### What is the RAM Model?

> [!info] Definition
> The **Random Access Machine (RAM) model** is a theoretical framework used to analyse and compare algorithms independently of the hardware they run on.

The RAM model focuses on the logical steps an algorithm performs rather than the characteristics of a specific computer.

### Why Use the RAM Model?

Testing algorithms on real hardware can produce inconsistent results because performance depends on factors such as:

- Processor speed
- Cache size
- Memory hierarchy
- Operating system
- Compiler optimisations
- Background processes

The RAM model removes these variables, allowing algorithms to be compared fairly.

### Assumptions of the RAM Model

| Assumption | Description |
| ---------- | ----------- |
| Infinite Memory | Memory is assumed to never run out. |
| Uniform Memory Access | Every memory location can be accessed in **O(1)** time. |
| Sequential Execution | Instructions execute one after another. |
| Constant-Time Operations | Arithmetic operations and comparisons each take **O(1)** time. |
| Simple Instruction Set | Only basic operations such as load, store, arithmetic and branching are considered. |

### Benefits

- Hardware-independent algorithm analysis.
- Provides a universal method for comparing algorithms.
- Enables theoretical analysis using **Big-O notation**.
- Produces consistent and reproducible results.

### Limitations

> [!warning]
> The RAM model is a simplified representation of a computer and does not accurately reflect real hardware.

Real computers include:

- Cache hierarchies
- Limited memory
- Multi-core processors
- Different memory access times

As a result, real-world performance may differ from theoretical analysis.

### RAM Model vs Real Computers

| RAM Model | Real Computer |
| ---------- | ------------- |
| Infinite memory | Limited memory |
| Uniform memory access | Cache and memory hierarchy |
| Sequential execution | Parallel and multi-core execution |
| Constant-time operations | Execution time varies by operation and hardware |

### Summary

> [!summary]
>
> - The RAM model is a theoretical framework used to analyse algorithms.
> - It removes hardware-specific factors, allowing algorithms to be compared fairly.
> - It assumes constant-time memory access and constant-time basic operations.
> - It provides the foundation for analysing algorithms using **Big O notation**.
> - Real computers differ from the RAM model due to hardware constraints such as caches and memory hierarchies.

## Big O Notation & Growth Rates

### Overview

> [!info] Definition
> The **growth rate** of an algorithm describes how its runtime increases as the size of the input (**n**) grows. It is used to classify an algorithm's efficiency and scalability.

As the input size increases, the focus shifts from the exact execution time to **how quickly the runtime grows**.

### Common Growth Rates

| Growth Rate | Big O | Efficiency | Example |
| ------------ | :---: | ---------- | ------- |
| Constant | **O(1)** | ⭐⭐⭐⭐⭐ | Array index lookup |
| Logarithmic | **O(log n)** | ⭐⭐⭐⭐⭐ | Binary Search |
| Linear | **O(n)** | ⭐⭐⭐⭐ | Linear Search |
| Log-Linear | **O(n log n)** | ⭐⭐⭐⭐ | Merge Sort, Heap Sort |
| Quadratic | **O(n²)** | ⭐⭐ | Selection Sort, Insertion Sort |
| Cubic | **O(n³)** | ⭐ | Triple nested loops |
| Exponential | **O(2ⁿ)** | ❌ | Brute-force subset generation |
| Factorial | **O(n!)** | ❌ | Brute-force permutations |

### Why Growth Rates Matter

> [!note]
> Algorithms with poor growth rates may perform well on **small datasets**, but become impractical as the input size increases.

Examples:

- **O(1)** remains almost unchanged regardless of input size.
- **O(log n)** grows very slowly.
- **O(n)** scales proportionally with the input.
- **O(n²)** and **O(n³)** become significantly slower for large datasets.
- **O(2ⁿ)** and **O(n!)** become impractical even for relatively small inputs.

### Practical Interpretation

| Input Size | Efficient Algorithms | Inefficient Algorithms |
| ---------: | -------------------- | ---------------------- |
| Small (8–100) | Most algorithms perform acceptably. | Differences are often negligible. |
| Medium (1,000–1,000,000) | O(1), O(log n), O(n), O(n log n) remain practical. | O(n²) and above become increasingly expensive. |
| Large (1 billion+) | Only low growth-rate algorithms remain practical. | O(n²), O(2ⁿ), and O(n!) become effectively unusable. |

### Summary

> [!summary]
>
> - Big O notation describes how an algorithm scales as the input size increases.
> - Lower growth rates generally produce more efficient algorithms.
> - **O(1)** and **O(log n)** are highly efficient.
> - **O(n)** and **O(n log n)** are suitable for most large-scale problems.
> - **O(n²)** and **O(n³)** become expensive as datasets grow.
> - **O(2ⁿ)** and **O(n!)** become impractical very quickly.
> - Choosing algorithms with lower growth rates is essential for building scalable software.
